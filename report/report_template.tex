\section{Description of the Problem}
This project focuses on the performance optimization of the Gaussian Mixture Model (GMM) training process. GMM is a robust probabilistic model used extensively in machine learning, signal processing, and data analysis for tasks such as clustering, anomaly detection, and density estimation. Training a GMM entails fitting a dataset of $N$ points in a $D$-dimensional space to $K$ distinct Gaussian distributions by estimating their corresponding weights, means, and covariance matrices.

The parameter estimation is heavily reliant on the iterative Expectation-Maximization (EM) algorithm, which alternates between two computationally rigorous phases:
\begin{itemize}
    \item \textbf{E-Step (Expectation):} Evaluates the multivariate Gaussian probability density function for every data point across all $K$ clusters to compute posterior probabilities (responsibilities).
    \item \textbf{M-Step (Maximization):} Aggregates the responsibilities across the dataset to update the model parameters (weights, means, and covariances).
\end{itemize}

As dataset sizes ($N$) and dimensionalities ($D$) increase, the EM algorithm becomes highly computationally and memory intensive due to continuous data passes and complex linear algebra operations (such as Cholesky decompositions for covariance matrix inversion). The primary objective of this project is to analyze the performance bottlenecks of a serial GMM implementation and accelerate the algorithm by utilizing parallel computing paradigms, specifically OpenMP for multi-core CPUs and CUDA for GPU architectures.

\section{Serial Code and Algorithm}
The base serial implementation fundamentally loops through a predefined number of iterations or until convergence, executing the Expectation (E) and Maximization (M) steps sequentially. 

\begin{itemize}
    \item \textbf{Expectation Step:} For every data point $x_i$ ($1 \le i \le N$) and every mixture component $k$ ($1 \le k \le K$), the algorithm computes the continuous multidimensional Gaussian probability. This involves first performing a Cholesky decomposition to invert the covariance matrix, then computing the Mahalanobis distance by subtracting the mean vector, multiplying by the inverse covariance matrix, and calculating the exponential. The results are normalized to determine the responsibility $\gamma_{i,k}$, representing the probabilistic assignment of point $x_i$ to cluster $k$.
    \item \textbf{Maximization Step:} The responsibilities are used to recalculate the parameters of the distributions. First, the algorithm computes the total effective weight for each cluster. Next, it performs a full sweep through the dataset to compute the new mean vectors. Finally, it executes another sweep to update the covariance matrices using the newly found means and the responsibility weights.
\end{itemize}

\textbf{Algorithmic Complexity:} The computational complexity per EM iteration is driven by varying operations depending on the dataset grid. During the E-step, computing the multivariate Gaussian PDF requires a Cholesky decomposition and subsequent matrix inversion of the covariance matrix for each of the $K$ clusters, an operation that scales dramatically as $\mathcal{O}(K \cdot D^3)$. Furthermore, computing the Mahalanobis distance for all $N$ points and updating the covariances in the M-step dictates vector-matrix multiplications scaling as $\mathcal{O}(N \cdot K \cdot D^2)$.

Therefore, for small dimensionality ($D \le 16$), the $\mathcal{O}(N \cdot K \cdot D^2)$ memory-bound operations dominate the execution. As $D$ increases, the $\mathcal{O}(K \cdot D^3)$ Cholesky matrix inversions cause the workload to become immensely compute-bound, representing the hardware inflection point between CPU caching/vectorization advantages and raw GPU FLOP throughput. Over $I$ iterations, the total computational complexity is bounded by $\mathcal{O}(I \cdot (K \cdot D^3 + N \cdot K \cdot D^2))$.

\section{Performance Profiling \& Bottlenecks}
Performance profiling on the serial execution (using tools like \texttt{gprof} or Xcode Instruments) reveals that the overwhelming majority of the runtime is concentrated within two heavily nested loops: the Gaussian probability calculation in the E-step, and the covariance matrix calculation in the M-step. 

When evaluating the arithmetic intensity (the ratio of floating-point operations to bytes of memory accessed), GMM training is notoriously \textbf{memory-bandwidth bound} rather than strictly compute-bound. 

For every iteration:
\begin{enumerate}
    \item The E-step reads the entire $N \times D$ dataset matrix.
    \item The M-step (means calculation) reads the $N \times D$ dataset matrix again along with the $N \times K$ responsibilities.
    \item The M-step (covariance calculation) reads the $N \times D$ dataset matrix a third time to calculate the outer products.
\end{enumerate}

This data-access paradigm forces the processor to stream the massive $N \times D$ memory footprint from main DRAM three separate times per iteration. Consequently, parallel optimization strategies must account for memory movement, as compute pipelines often stall waiting for variable fetches rather than arithmetic exhaustion.

\section{Data Structures \& Memory}
To support the EM algorithm, the implementation relies on several primary data structures. To ensure cache locality and avoid pointer chasing, these are allocated as contiguous 1D memory blocks representing 2D or 3D matrices in row-major order:
\begin{itemize}
    \item \textbf{Dataset (\texttt{data}):} An $N \times D$ matrix containing the input features.
    \item \textbf{Means (\texttt{means}):} A $K \times D$ matrix tracking the centroid of each Gaussian component.
    \item \textbf{Covariances (\texttt{covariances}):} A $K \times D \times D$ tensor representing the covariance matrix for each component.
    \item \textbf{Responsibilities (\texttt{responsibilities}):} An $N \times K$ matrix storing the probability of each point belonging to each class.
    \item \textbf{Inverse Covariances:} A pre-calculated $K \times D \times D$ tensor to avoid repeated matrix inversions inside the inner loop.
\end{itemize}

\textbf{Memory Reference Pattern:} The memory access pattern across the dataset is largely sequential. During the E and M-steps, the outermost loop iterates over the dataset $N$, and the inner loops iterate over $K$ and $D$. Because the dataset is laid out such that the $D$ dimensions for a single point are perfectly contiguous, the CPU accesses the memory sequentially, step-by-step. 

This sequential read pattern is highly cache-friendly. Modern CPU hardware prefetchers can easily detect the linear access stride and proactively load the memory lines into L1 and L2 caches ahead of the execution core, efficiently masking latency. However, for extremely large values of $N$, the full dataset capacity easily exceeds the L3 cache, causing unavoidable capacities misses and cyclic cache flushing over the iterations.

\section{Parallel Modifications}
To transition the serial algorithm into parallel applications capable of producing mathematically identical results asynchronously, several macro structural modifications were enacted to resolve deep-seated data dependencies and prevent race conditions.

\begin{itemize}
    \item \textbf{OpenMP (CPU):} In the baseline serial code, updating the global distribution parameters (means and covariances) during the M-step causes massive data contention. To safely parallelize this, thread-local storage was introduced. Every active thread allocates a private computational space (e.g., \texttt{local\_weight}, \texttt{local\_mean}, and \texttt{local\_covariance}). Threads autonomously accumulate partial sums from their designated data chunks into these private structures lock-free. After the intensive parallel loops conclude, a singular \texttt{\#pragma omp critical} syncing step securely aggregates the scattered local results into the unified global distribution parameters.
    \item \textbf{CUDA (GPU):} The GPU adaptation requires translating deeply nested iteration loops into flattened kernel launches. Instead of looping through instances sequentially, the outer dataset loop iterating over $N$ is mapped to the GPU grid layout. Each CUDA thread calculates the probabilities and continuous responsibilities exclusively for a single assigned data point. To update the global parameters without memory corruption, the kernels rely on \texttt{atomicAdd()} operations to serialize concurrent floating-point additions effectively.
\end{itemize}

\section{Partitioning Strategy}
Determining how the immense data workload is distributed directly influences the efficiency of the implementation.

\begin{itemize}
    \item \textbf{OpenMP Partitioning:} Since the algorithmic iterations over the $N$ points in the dataset are operationally independent during the E-step and early M-step, a 1D flat partitioning approach is utilized. By defining \texttt{\#pragma omp parallel for schedule(static)}, the $N \times D$ data matrix is evenly sectioned out directly to the active hardware threads. For an execution configuration of $N=500,000$ points running across 24 threads, each underlying core is assigned to process precisely $\approx 20,833$ contiguous points, ensuring absolute L1 cache lineage continuity.
    \item \textbf{CUDA Partitioning:} The CUDA grid is similarly modeled leveraging 1D mapping logic. The absolute thread index (\texttt{tid = blockIdx.x * blockDim.x + threadIdx.x}) acts directly as the designated pointer to an individual point $x_i$. Grid dimensions are dynamically evaluated using ceiling integer division (e.g., $\lceil N / 256 \rceil$), sorting the workload into parallel thread-blocks of size 256. This layout forces Streaming Multiprocessors to access the contiguous feature arrays in memory-coalesced 32-thread warp executions.
\end{itemize}

\section{Optimizations Overview}
Once parallel correctness was achieved, several underlying hardware optimizations were layered onto the structure to enhance memory bandwidth utilization and floating-point throughput.

\textbf{OpenMP CPU Optimizations:}
\begin{enumerate}
    \item \textbf{SIMD Vectorization Alignments:} The defined mathematical baseline ($D=32$) acts as a highly optimized parameter metric for contemporary CPUs. Since 32 floating-point values resolve to exactly 128 bytes, they efficiently saturate AVX2 (256-bit) and AVX-512 vector lengths across execution cycles. Utilizing the `gcc` compiler flags (`-O3 -ffast-math`), innermost $D$ dimension loops natively compile down to singular SIMD vectorized assembly instructions without requiring manual intrinsic injections.
    \item \textbf{Lock-Free Execution Cycles:} Replacing persistent reduction locks or atomic operations inside the loop body with the isolated `local\_covariance` arrays bypasses immense OS thread-contention and lock latency across the 24 cores.
\end{enumerate}

\textbf{CUDA GPU Optimizations \& Challenges:}
\begin{enumerate}
    \item \textbf{Shared Memory Tiling Mechanisms:} To deflect the severe latency involved in repeatedly accessing configuration tensors (like the \texttt{means} array) directly from standard HBM2 global memory, evaluating blocks proactively load subsets of cluster constraints directly into local SM \texttt{\_\_shared\_\_} memory during initial execution stages, streamlining execution loops directly from SRAM.
    \item \textbf{Challenge: Heavy Atomic Contention:} During the CUDA M-step reduction phases, compiling the global matrices requires significant executions of \texttt{atomicAdd} in global or shared memory. In parameter ranges operating high cluster counts ($K$), increasing volumes of threads attempt to write over the exact identical memory destinations simultaneously for probability bins. This hardware bottleneck mandates immense thread serialization queuing logic, occasionally forcing the GPU implementation to underperform its highly vectored CPU equivalent.
\end{enumerate}

\section{Experiments and Results}
In this section, we evaluate the performance of our serial, OpenMP, and CUDA implementations across a broad parameter grid. The experiments were conducted on a dataset of $N=500,000$ points, varying the dimensionality ($D$) from 4 to 32 and the number of clusters ($K$) from 2 to 96.

\subsection{CPU vs GPU Scaling Analysis}
Our benchmarking results reveal a significant performance "cross-over point" between the 32-thread OpenMP CPU implementation and the CUDA GPU implementation (Tesla P100). As shown in the experimental data, the optimal hardware choice depends strictly on the computational complexity of the GMM configuration.

\begin{itemize}
    \item \textbf{Low Complexity and Overhead:} For small values of $K$ and $D$, the OpenMP implementation consistently outperforms CUDA. For instance, at $D=4, K=2$, OpenMP completes in 0.011s compared to CUDA's 0.098s. This is primarily due to the fixed overhead of CUDA kernel launches and the fact that the small model fits entirely within the CPU's L2/L3 caches.
    \item \textbf{The Cross-over Point:} As $D$ and $K$ increase, the workload transitions from being memory-latency bound to compute-throughput bound. 
    \begin{itemize}
        \item For $D=8$, the cross-over occurs at $K=8$, where CUDA begins to lead.
        \item For $D=32$, OpenMP holds its advantage until $K \approx 12$, after which CUDA's massive parallelism takes over.
    \end{itemize}
    \item \textbf{High Complexity Scaling:} At high cluster counts ($K \ge 32$), the GPU's superior throughput becomes dominant. At the maximum tested configuration ($D=32, K=96$), the CUDA implementation achieves a wall time of \textbf{9.3s}, outperforming the 32-thread OpenMP implementation (\textbf{25.5s}) by a factor of \textbf{2.7x}, and the serial reference (\textbf{88.8s}) by nearly \textbf{10x}.
\end{itemize}

\subsection{Architectural Performance Factors}
The observed scaling behavior is driven by several competing architectural factors:
\begin{itemize}
    \item \textbf{Arithmetic Intensity:} At low $K$, the ratio of computation to memory access is low, making the GPU's memory bandwidth and synchronization overhead the limiting factors. As $K$ increases, the $O(N \cdot K \cdot D^2)$ operations dominate, allowing the GPU to hide memory latency through massive thread concurrency.
    \item \textbf{CPU Vectorization Efficiency:} For $D=32$, the CPU utilizes AVX-512 instructions to process the dimensionality loop with extreme efficiency. This keeps the CPU competitive even at moderate complexities.
    \item \textbf{Atomic Contention:} In the CUDA M-Step, `atomicAdd` operations face high contention at low $K$ as many threads attempt to update the same cluster parameters. As $K$ increases, these updates are spread across more memory addresses, paradoxically reducing contention and improving the GPU's scaling efficiency.
\end{itemize}

\section{Future Work for GPU Acceleration}
To further optimize the CUDA implementation and move the cross-over point to lower complexities, the following architectural redesigns are proposed:
\begin{enumerate}
    \item \textbf{Kernel Fusion:} Combining the E-step and M-step kernels would reduce the number of passes over the $N \times D$ dataset from three to one, significantly easing the memory bandwidth bottleneck.
    \item \textbf{GPU-Native Cholesky:} Offloading the Cholesky decomposition to cuSOLVER or a custom block-level kernel would eliminate the expensive iteration-blocking \texttt{cudaMemcpy} calls and CPU-GPU synchronization.
    \item \textbf{Warp-Level Reductions:} Utilizing warp-shuffle instructions (\texttt{\_\_shfl\_down\_sync}) for local reductions within the Streaming Multiprocessor would eliminate the reliance on \texttt{atomicAdd} in shared memory, further boosting throughput in the M-Step.
\end{enumerate}

\section{Conclusion}
This project successfully demonstrated the fundamental principles determining High-Performance Computing (HPC) outcomes when porting complex machine learning models to parallel architectures. By implementing the Expectation-Maximization algorithm for Gaussian Mixture Models in both OpenMP and CUDA, critical intuitions surrounding hardware bottlenecks were quantified.

What worked exceptionally well was the OpenMP thread-local caching strategy combined with AVX-512 vector pipeline unrolling, which allowed the 32-thread CPU implementation to completely dominate memory-bound, low-complexity models (low $D$, low $K$). The L3 cache capabilities effectively masked main DRAM latencies in a way the GPU's PCIe bus transfers could not.

However, grid testing revealed absolute architectural limits. Once the dimensionality scalar forced expensive $\mathcal{O}(K \cdot D^3)$ Cholesky matrix inversions and massive cluster reductions simultaneously, the CPU pipelines were overwhelmed. Here, the raw floating-point strength of the Tesla P100 GPU breached the cross-over point. Despite heavy \texttt{atomicAdd} contention in shared memory and synchronization limits, the GPU delivered up to a $10\times$ speedup over the serial implementation under maximum tested stress limits ($D=32, K=96$).

Ultimately, this optimization exercise emphasized that parallel strategy cannot be one-size-fits-all. Crossover thresholds dictate the correct hardware target. Moving forward, eliminating memory movement penalties through advanced GPU Kernel Fusion and offloading Cholesky math to \texttt{cuSOLVER} remain the clearest paths to maximizing strict GPU supremacy across all model sizes.
